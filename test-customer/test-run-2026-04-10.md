## x402 Payment Flow

The storefront uses x402 V2 protocol for payment gating:

1. **Initial request** → server returns HTTP 402 with `x-payment-required` header (base64-encoded JSON)
2. **Decode payment requirements:**
   ```json
   {
     "scheme": "exact",
     "network": "eip155:8453",
     "asset": "0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913",
     "amount": "10000",
     "payTo": "0xFEB27852DAc5a9449B0fECf4a1D98Ecf76F0c24B",
     "maxTimeoutSeconds": 300,
     "extra": { "name": "USD Coin", "version": "2" }
   }
   ```
3. **Sign EIP-712 TransferWithAuthorization** — domain: `{ name: "USD Coin", version: "2", chainId: 8453, verifyingContract: <USDC> }`
4. **Retry with `PAYMENT-SIGNATURE` header** — value is base64-encoded JSON:
   ```json
   {
     "x402Version": 2,
     "payload": { "authorization": { ... }, "signature": "0x..." },
     "accepted": { ... }
   }
   ```

**Key implementation note:** The `PAYMENT-SIGNATURE` header value must be base64-encoded (not raw JSON). The `@x402/fetch` library handles this; manual implementations must apply `Buffer.from(JSON.stringify(payload)).toString("base64")`.

## Issues Found

1. **Query endpoint returns 500** — `GET /data/agent-systems.csv/query?scheduling_cron=True` returns `{"detail":"Internal server error"}` after payment is accepted. The filtered-query feature is broken server-side; the download endpoint works correctly.

2. **No `x-payment-receipt` header** — Neither the download nor tip responses returned an `x-payment-receipt` or `X-Payment-Receipt` header. Receipts would be useful for buyers to verify on-chain settlement.
